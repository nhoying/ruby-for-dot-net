Declaring Methods

To declare a method you can use the def keyword 

                def a_really_cool_method(some_parameters)
                  # some really cool code
                end
If you don't have any parameters you can leave the () out 

                def where_were_going_we_dont_need_params
                  # some really cool code
                end
The same applies for when calling the method 

                # uses a parameter
                parameter = "some value"
                a_really_cool_method(parameter)

                # doesn't use a parameter
                where_were_going_we_dont_need_params

Blocks

One of the most powerful things about Ruby are blocks. They are anonymous functions that can be passed into methods, kinda like a C# Lambda function.
Think of the following C# code: 

                List myList = new List() [ 1, 2, 3 ];
                
                myList.ForEach(a => Console.WriteLine(a));
This could be written in Ruby as 

                my_list = [ 1, 2, 3 ]
                
                my_list.each | a | do
                  puts a
                end
If your code only takes a single line, you can also use a shorthand syntax: 

                my_list = [ 1, 2, 3 ]
                
                my_list.each {| a | puts a }
I'm just scratching the surface on blocks. There's a lot of cool stuff you can do with them!

Migrations and Schema

Migrations are created from the command line.

This will generate a blank migration 


                rails g migration MyMigration

Depending how you name your migration, you can specify the different columns on the command line as well.

This will generate a migration that adds a column named part_number to the parts table. 


                rails g migration AddPartNumberToParts part_number:string

It will generate a file in the db/migrate folder called timestamp_your_migration_name.rb 


                class AddPartNumberToParts < ActiveRecord::Migration[6.0]
                  def change
                    add_column :parts, :part_number, :string
                    add_index :parts, :part_number
                  end
                end

As part of the Convention over Configuration principle, Rails will automatically create the necessary primary and foreign key columns as well as created and modified timestamp columns to your table.

Working with ActiveRecord

In order to access your database after creating your schema, you'll need to create a model. You can combine this with your migration with the following rails console command: 

                rails g model Part part_number:string name:string
In addition to the migration, it will also create a file in your app/models folder. 

                class Part < ApplicationRecord
                end

Notice how there are no accessors or properties defined. This is all implicit in Ruby on Rails.

You can select a number of ways off of the ActiveRecord model class. 


              # Returns all records
              Part.all
            

              # retrieves the first record based on the passed criteria
              Part.find_by(part_number: 'A-123456')
            

              # retrieves a collection of records based on the passed critiera
              Part.where(part_number: 'A-123456')

There are also a number of ways you can write to the database. 


              # Creates a record
              Part.create(part_number: 'A-123456', name: 'Small Widget')
            

              # Update a record
              part.update(name: 'Smaller Widget')
            

              # You can also manipulate the returned Part object and call save on it
              part.name = 'Smaller Widget'
              part.save

The Rails MVC Engine

Thankfully, there is not a whole lot of difference between ASP.NET and Ruby on Rails's implementation of the MVC (model/view/controller architecture). There are still controllers. There are still views. There are still models.

Routing

Routing is configured in the config/routes.rb file 

              Rails.application.routes.draw do
                root 'landingpage#home'
                resources :hardwares, path: 'hardware'
                resources :overviews, only: [:show, :update, :destroy]
                # For details on the DSL available within this file, see https://guides.rubyonrails.org/routing.html
              end
Resource routing allows us to quickly declare common routes for a given controller.
  • index
  • show
  • new
  • edit
  • create
  • update
  • destroy
 You can use the only keyword to limit the available routes

Controllers

Controllers work in much the same way as they do in ASP.NET There are some key differences though!
Much of the boilerplate work for a controller has been taken care of for you. 

              // aspnetCore
              public IActionResult New()
              {
                return View();
              }
              
              # Ruby on Rails
              def new
              end
Parameters are passed through an object instead of being defined as parameters 

              // aspnetCore
              public IActionResult Show(int id)
              {
                model = GetModel(id);
                return View(model);
              }
              
              # Ruby on Rails
              def show
                SomeModel.find(id: params[:id])
              end

Views

Views also work in much the same way as they do in ASP.NET. The biggest difference is syntax.

Testing and RSpec

There are a number of testing frameworks such as minitest, test-unit (similar to xunit), and cucumber. The one I'm most familar with (because this is what they use at my client) is rspec.

The describe block

The describe block quite literally describes the piece of code you are testing.
Usually you'll start with a describe block for your class 

                  describe MyModule::MyClass do
                    # Test Code
                  end
Then you'll add describe blocks for your methods inside of the main describe block 

                  describe MyModule::MyClass do
                    describe '#my_method' do
                      # test code for my method
                    end
                  end

The context block

The context block then allows you to specify a scenario for your test These need to start with a keyword like when 

                  describe MyModule::MyClass do
                    describe '#my_method' do
                      context 'when my_parameter is set to 2' do
                        # test code for the context
                      end
                    end
                  end
You can add as many context blocks as you'd like 

                  describe MyModule::MyClass do
                    describe '#my_method' do
                      context 'when my_parameter is set to 2' do
                        # test code for the context
                      end

                      context 'when my_parameter is set to 8' do
                        # test code for the context
                      end

                      context 'when my_parameter is set to 9' do
                        # test code for the context
                      end
                    end
                  end

The it block

The it block then sets up the expectation for the method
These also need to start with a keyword like returns 

                  describe MyModule::MyClass do
                    describe '#my_method' do
                      context 'when my_parameter is set to 2' do
                        it 'returns 4' do
                          expect(described_class.my_method(2)).to eq(4)
                        end
                      end
                    end
                  end

Defining your test data with the let method

let allows you to set a global block that you can use to help define your test objects 

                  let(:my_big_test_data) do
                    {
                      property_1: 'Hardcoded',
                      property_2: my_test_data,
                    }
                  end
                  let(:my_test_data) { 'some value' }
You can use a subsequent let on the same symbol to change your data downstream without having to re-declare anything that references it 

                  let(:my_big_test_data) do
                    {
                      property_1: 'Hardcoded',
                      property_2: my_test_data,
                    }
                  end
                  let(:my_test_data) { 'some value' }
                  context 'when this happens' do
                    it 'returns some value' do
                      expect(described_class.my_method(my_big_test_data)).to eq('some value')
                    end
                  end

                  context 'when that happens' do
                    let(:my_test_data) { 'some other value'}
                    it 'returns some other value' do
                      expect(described_class.my_method(my_big_test_data)).to eq('some other value')
                    end
                  end

Mocking methods with the allow method

allow is similar to setting up a C# interface to return something using the Moq framework. 

                allow(MyMockedClass)
                  .to receive(:my_method_name)
                  .with(my_parameter)
                  .and_return(my_result)
